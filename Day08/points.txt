Schedule a pipeline
--------------------
1. Time based scheduling
   The most common method it runs workflows at fixed intervals regardless of the 
   external activity.

2. Event based scheduling
   Instead of waiting for a clock to strike these pipelines wait for a specific external 
   event or signal to occur.

3. Dependency based scheduling
   Pipelines are triggered as part of a larger interconnected workflow (often called DAG 
   Directed Acyclic Graph).

4. Hybrid scheduling
   Combines time based triggers with conditional logic or rules.

5. On demand scheduling
   Not a automated schedule but a core method of running pipelines.

Most of the scheduling is done the cloud environment.

CI / CD
--------
Continuous Integration / Continuous Deployment (Delivery)

CI -> is the automated process of frequently merging code changes from multiple developers into a shared repository

Every time the code is saved and pushed to a version control system (GitHub, GitLab) automated system builds the application and run tests

It prevents integration nightmares by caching bugs conflicts and error immediately ensuring the codebase is always stable

CD -> takes over right after CI finishes. It comes in two variations

Delivery -> Code changes are automatically built, tested and prepped for a release. However the final push to the production requires a manual click or approval by a human.

Deployment -> Every code changes that passes the automated tests goes through the entire pipeline and is automatically deployed to the production environment(to end users) without any human intervention.

CI Workflow
-----------

Developer write and commits the code
CI tools build the application
Automated tests are executed
if issue occurs -> 'Problem is detected' -> developer fixes the code
if successful ->'Everything OK' -> code is merged
Application is ready for deployment
